# M4 产品流程重设计 — Photo-to-Pixel Customization

> 文档版本：v1 | 创建日期：2026-07-01 | 作者：产品经理许清楚
> 状态：待架构师评审技术路线

---

## 产品目标

一句话定义 M4 核心价值：

**玩家上传人物照片 → 系统提取人物特征 → 重塑为符合像素规范的小人 → 作为桌宠角色在桌面上生活**

不再只是"给模板换个颜色"，而是真正从照片中**识别人物本身**并**重塑**为像素小人。

---

## 新旧流程对比

| 步骤 | 旧流程 | 新流程 |
|------|--------|--------|
| 上传照片 | 选择本地图片 | 选择本地图片（不变） |
| 人物提取 | ❌ 只取整张图的平均色 | ✓ 从照片中分割出人物本身（去除背景） |
| 特征识别 | ❌ 只有整体平均色 | ✓ 分别识别头发色、肤色、衣服色、配饰色 |
| 像素重塑 | ❌ 将颜色涂到固定的同一模板 | ✓ 根据人物特征选择发型、服装、配饰等模板部件 |
| 默认角色 | 固定模板作为唯一产出 | 占位资源（玩家上传前的临时显示） |
| 最终产出 | 换颜色的同一张丑脸 | 根据照片特征重塑的个性化像素小人 |

## 核心机制：特征→像素映射规则

这是本次重设计的最关键部分。定义从照片中提取哪些特征，以及每个特征如何映射到 48×64 像素模板的绘制。

### 区域分割策略

照片被分割为 5 个采样区域（基于人物分割后的 bounding box）：

```
┌───────────────────┐
│   顶部：头发区域    │ ← y: 0% ~ 25%
│                   │
├───────────────────┤
│   中上：面部区域    │ ← y: 25% ~ 45%
│                   │
├───────────────────┤
│   中部：上衣区域    │ ← y: 45% ~ 70%
│                   │
├───────────────────┤
│   下部：下装区域    │ ← y: 70% ~ 85%
│                   │
├───────────────────┤
│   底部：鞋子/脚     │ ← y: 85% ~ 100%
└───────────────────┘
```

### 特征提取与映射表

| 照片特征 | 提取方式 | 像素映射 | 优先级 |
|----------|----------|----------|--------|
| **头发颜色** | 从照片顶部区域（y: 0%~25%）采样主色 | `palette.hair` → 所有发型函数的主色 | P0 |
| **头发长度** | 从分割结果计算头发区域的高度占比（占头部 bounding box 的比例） | 选择发型函数：短发→`hairShort()` / 中发→`hairBob()` / 长发→`hairLong()` / 卷发→`hairCurly()` | P0 |
| **皮肤颜色** | 从照片中上面部区域（y: 25%~45%）采样主色 | `palette.body` + `head()` 的肤色（替换硬编码的 `#ffe6d2`） | P0 |
| **上衣颜色** | 从照片中部区域（y: 45%~70%）采样主色 | `palette.outfit`（主色）+ `palette.accent`（次色/装饰色） | P0 |
| **下装颜色** | 从照片下部区域（y: 70%~85%）采样主色 | `shoes()` 颜色 + 下装区域色（当前模板中下装与上衣同色系，新流程可独立设色） | P1 |
| **配饰检测** | 检测是否有耳机/帽子等明显配饰（颜色块在头部边缘） | 选择是否调用 `headphones()` / 帽子函数 | P1 |
| **身体比例** | 从分割结果计算头身比（头部占比 vs 身体占比） | 调整模板头部/身体的像素比例（默认 Q 版 2.5 头身，可微调至 2~3 头身） | P2 |

### 颜色采样规则

不再使用 `averageImageColor()` 取全图平均色。新方案采用**区域加权主色提取**：

1. 对分割后的人物区域（去除背景），按上述 5 个采样区域裁切
2. 每个区域独立做颜色聚类（简易 k-means 或频率排序，取 top-2 颜色）
3. 主色 → 该区域的主要绘制色
4. 次色 → 该区域的装饰/条纹色（如有）
5. 所有颜色经过 `quantize()` 量化到 24 步进，保证在 8~16 色调色板内不溢出

## 发型变体设计（新增）

当前只有 `hairBob()`（鲍勃短发），需要新增至少 3 种发型以覆盖常见人物照片。

### 1. `hairShort()` — 短发/寸头

**适用场景**：男性常见发型，头发区域高度占比 ≤ 20%

**形状描述**：
- 头顶覆盖一层短帽状头发，紧贴头部轮廓
- 不向两侧延伸，无侧垂发片
- 适合配合 `headphones()` 或帽子等配饰

**大致坐标范围**（48×64 画布）：
```
外轮廓椭圆：centerX=24, centerY=15, radiusX=12, radiusY=9
内填充椭圆：centerX=24, centerY=16, radiusX=10, radiusY=7
头顶额外覆盖：rect(16, 7, 17, 3)  ← 加厚顶部
无侧垂发片（与 hairBob 的关键区别）
```

### 2. `hairLong()` — 长发披肩

**适用场景**：女性常见发型，头发区域高度占比 ≥ 35%

**形状描述**：
- 头顶有蓬松发顶，两侧有长发垂至肩膀以下
- 侧垂发片延伸至腰部区域（比 hairBob 更长）
- 发片底部可以略微内收或微弧

**大致坐标范围**（48×64 画布）：
```
外轮廓椭圆：centerX=24, centerY=15, radiusX=14, radiusY=13
内填充椭圆：centerX=24, centerY=16, radiusX=12, radiusY=11
头顶覆盖：rect(17, 8, 15, 6)
左侧垂发：rect(10, 17, 5, 22)  ← 从肩膀延伸到腰部
右侧垂发：rect(34, 17, 5, 22)
阴影区：rect(21, 14, 3, 2) + rect(26, 14, 3, 2)
```

### 3. `hairMedium()` — 中长发（当前 `hairBob()` 即可归入此类）

**适用场景**：头发区域高度占比 20%~35%，当前鲍勃短发覆盖此区间

**形状描述**：维持现有 `hairBob()` 的设计，侧垂发片到肩膀中部

**无需新增函数**，当前 `hairBob()` 直接作为中长发变体使用。后续如有需要可微调侧垂长度。

### 4. `hairCurly()` — 卷发/蓬松发型

**适用场景**：头发区域横向占比大、轮廓不规则

**形状描述**：
- 头顶蓬松隆起，横向比其他发型更宽
- 两侧有卷曲发团（用多个小椭圆堆叠模拟卷曲感）
- 整体轮廓比其他发型更圆润宽大

**大致坐标范围**（48×64 画布）：
```
外轮廓椭圆：centerX=24, centerY=14, radiusX=16, radiusY=12
内填充椭圆：centerX=24, centerY=15, radiusX=14, radiusY=10
左侧卷发团：ellipse(centerX=10, centerY=20, radiusX=4, radiusY=5)
右侧卷发团：ellipse(centerX=38, centerY=20, radiusX=4, radiusY=5)
头顶隆起：rect(14, 5, 21, 5)
阴影区：rect(20, 14, 4, 2) + rect(26, 14, 4, 2)
```

### 发型选择逻辑

```
头发区域高度占比 ≤ 20%  → hairShort()
头发区域高度占比 20~35% → hairBob()（中长发）
头发区域高度占比 ≥ 35%  → hairLong()
头发区域横向占比异常大  → hairCurly()
```

> 注：头发区域占比的计算依赖人物分割模型，占比阈值需在实机测试中微调。

## 服装变体设计

### 当前已有变体

| 变体 ID | 绘制函数 | 服装类型 | 视觉描述 |
|---------|---------|---------|---------|
| `approved_base` | `paintApprovedBase` | 蓝白制服裙 | 上衣+裙装，有耳机 |
| `simple_dress` | `paintSimpleDress` | 简约连衣裙 | 一体裙装，无耳机 |
| `shorts_casual` | `paintShortsCasual` | 休闲短裤 | 上衣+短裤，有耳机 |
| `hoodie_casual` | `paintHoodieCasual` | 卫衣休闲 | 宽松卫衣+裤装 |
| `soft_uniform` | `paintSoftUniform` | 柔软制服 | 制服上衣+裙装 |

### 新流程中的服装选择逻辑

旧流程中，variantId 是固定传入的，不根据照片内容选择。新流程应根据照片中的服装特征自动匹配：

```
检测到裙装/连衣裙特征（下部为整体色块）  → simple_dress
检测到短裤/分体下装特征                → shorts_casual
检测到宽松外套/卫衣特征（轮廓宽大）      → hoodie_casual
检测到制服/合身上衣特征               → approved_base 或 soft_uniform
无法明确判断                           → approved_base（默认兜底）
```

> 注：服装类型的判断来自中部+下部区域的主色分布。裙装表现为下部单一色块，分体装表现为上下不同色。这个判断不需要 ML 模型，纯颜色区域分析即可。

## 占位角色设计

### 设计原则

默认角色只是一个**简单的占位资源**，玩家上传照片前随便显示一个即可。不需要特别精致，也不应该是唯一的产出模板。

### 推荐方案

使用当前的 `approved_base` 模板作为占位角色，搭配默认 `fallbackPalette`。理由：
- 已经有完整的绘制逻辑，不需要额外开发
- 视觉上足够清晰，能传达"这是一个像素小人"
- 玩家看到后会自然想要上传照片来替换它

**可选优化**：设计一个更简单的"空白小人"（只有最基本的头部+身体轮廓，无发型细节），进一步强化"等待自定义"的语义。但这是 P2 优先级，MVP 阶段直接用 approved_base 即可。

### 占位角色的文案提示

在桌宠界面靠近占位小人处，可显示一行提示文字：
- "点击 Import 上传照片，创建你的专属小人"
- 上传后提示文字消失，小人切换为自定义角色

## UX 交互流程

### 完整交互步骤

```
状态 1：初始状态
  → 桌面上显示占位小人（approved_base + 默认 palette）
  → 底部显示 "Import" 按钮
  → 可选：显示提示文案 "上传照片创建专属小人"

状态 2：玩家点击 Import
  → 弹出文件选择器（支持 PNG/JPEG/WebP/BMP/GIF）
  → 玩家选择一张人物照片

状态 3：系统处理
  → 显示处理中动画/进度指示（像素化 spinner 或文字 "正在分析..."）
  → 步骤：
    a) 人物分割：从照片中提取人物区域（去除背景）
    b) 区域采样：按 5 个区域分别采样颜色
    c) 特征识别：头发长度/肤色/上衣色/下装色/配饰
    d) 模板选择：根据特征选择发型函数 + 服装变体
    e) 像素绘制：用提取的 palette + 选定的部件函数绘制 48×64 像素小人

状态 4：预览显示
  → 预览框显示重塑后的像素小人
  → 下方显示特征识别摘要（可选）：
    "发型: 中长发 | 肤色: #ffe6d2 | 上衣: 蓝色 | 下装: 白色"
  → Save 按钮变为可点击

状态 5：确认/微调（可选功能，P1 优先级）
  → 玩家可切换发型变体（短发/中发/长发/卷发）
  → 玩家可切换服装变体（制服/连衣裙/休闲/卫衣）
  → 每次切换实时更新预览
  → 玩家确认满意后点击 Save

状态 6：保存并应用
  → 点击 Save → 生成 manifest.json + preview.png + art-metadata.json
  → 保存到 Electron userData/custom-characters/
  → 桌面上的小人切换为自定义角色
  → 底部 label 变为 "Custom Character"
  → 提示文案消失

状态 7：后续使用
  → 重启后自定义角色仍然激活（已有本地持久化）
  → 玩家可再次 Import 更换角色
```

### 微调界面的 UX 规范

微调功能为 P1 优先级（MVP 可不含），但产品流程中应预留：

- 发型切换：4 个缩略图标（短发/中发/长发/卷发），点击切换
- 服装切换：5 个缩略图标（对应已有变体），点击切换
- 切换后实时更新预览画布
- 不提供颜色手动调整（颜色应忠实反映照片，避免用户改出不符合像素规范的配色）

## 离线需求确认

**所有功能必须支持离线运行，不能依赖外部 API。**

具体要求：
- 人物分割模型必须本地加载（bundled in Electron app 或首次启动时从本地缓存加载）
- 颜色采样和特征识别全部在客户端 Canvas 中完成
- 不调用任何云端 API（图像识别、色名查询等）
- 不依赖网络下载模型文件
- 断网状态下 Import → Save 全流程正常完成

## 待确认问题（需架构师评估）

以下技术问题需要架构师在技术路线评审中回答：

### 1. 人物分割模型选择

| 方案 | 优点 | 缺点 | 离线支持 |
|------|------|------|---------|
| **BodyPix**（TensorFlow.js） | 成熟稳定，API 简洁，支持分段输出 | TF.js runtime ~3MB，BodyPix 模型 ~1.5MB | ✓ 完全离线 |
| **MediaPipe Selfie Segmentation** | 轻量快速，专为人物分割优化 | MediaPipe WASM 依赖较复杂 | ✓ 可离线但 WASM 加载需测试 |
| **纯 Canvas 假分割**（基于颜色聚类+边缘检测） | 0 依赖，bundle 零增量 | 精度低，对复杂背景/多人场景效果差 | ✓ 完全离线 |
| **ONNX Runtime + 轻量分割模型** | 模型可自定义，ONNX WASM 较小 | 需要额外集成 ONNX runtime | ✓ 完全离线 |

**产品建议**：优先评估 MediaPipe Selfie Segmentation（轻量+专用），其次评估 BodyPix。纯 Canvas 假分割作为兜底方案（如果模型大小不可接受）。

### 2. 模型大小对 Electron bundle 的影响

- 当前项目是否已有 TensorFlow.js 或 MediaPipe 依赖？
- 模型文件是否可以 lazy-load（首次 Import 时加载，非启动时）？
- 目标增量：bundle size 增加 < 5MB 为理想，< 10MB 为可接受

### 3. 首次加载时间是否可接受

- 模型首次加载预计耗时？（WASM 初始化 + 模型权重下载）
- 是否需要预加载/缓存机制？
- 用户等待时间 > 3s 时是否需要显示进度动画？

### 4. 是否需要预加载/缓存模型

- 模型在首次 Import 时加载并缓存到磁盘？
- 后续 Import 直接从磁盘缓存读取？
- 缓存位置：Electron userData 目录

### 5. 48×64 像素尺寸下的分割精度要求

- 48×64 的输出尺寸非常小，人物分割的精度需求是否可以降低？
- 是否可以先用低分辨率分割（如 96×128），再降采样到 48×64 用于区域分析？
- 低精度分割是否可以用纯算法替代 ML 模型？

### 6. 发型判断的可靠性

- 头发区域高度占比的阈值是否可靠？（不同照片角度、发型可能误判）
- 是否需要增加"用户手动选择发型"的兜底？
- 误判时的降级策略：默认 fallback 到 hairBob()

## 产品优先级（P0/P1/P2）

| 优先级 | 功能 | 说明 |
|--------|------|------|
| **P0** | 人物分割（去除背景） | 核心价值：不再对全图取平均色 |
| **P0** | 区域颜色采样（5 区域独立采样） | 核心价值：分别提取头发色/肤色/衣服色 |
| **P0** | 发型变体 + 自动选择 | 核心价值：小人不再永远是鲍勃头 |
| **P0** | 服装变体自动选择 | 核心价值：小人不再永远是同一套衣服 |
| **P0** | 离线全流程 | 约束条件：不能依赖网络 |
| **P1** | 微调界面（发型/服装切换） | 用户体验提升：允许玩家修正自动判断 |
| **P1** | 配饰检测（耳机/帽子） | 丰富性提升 |
| **P1** | 下装独立设色 | 丰富性提升 |
| **P1** | 处理中进度动画 | 用户体验提升 |
| **P2** | 身体比例微调 | 低价值：Q 版比例差异不大 |
| **P2** | 占位角色优化（空白小人） | 低价值：当前 approved_base 可用 |
| **P2** | 特征识别摘要文案 | 低价值：锦上添花 |

## 与现有代码的兼容性

### 保留不变的

- `PixelTemplatePalette` 类型定义（新增字段可选）
- `pixelTemplateSpec`（48×64 规格）
- 基础绘制工具函数 `rect()` / `ellipse()`
- 现有部件函数 `face()` / `head()` / `arms()` / `shoes()` / `headphones()`
- 所有现有服装绘制函数（`paintApprovedBase` 等）
- `generateTemplatePixelCharacter()` 的入口签名（内部逻辑重写）

### 需要重写的

- `paletteFromImage()` → 改为基于人物分割的区域采样
- `averageImageColor()` → 废弃，改为区域主色提取
- `hairBob()` → 不再是唯一发型，改为根据特征选择发型函数
- `paintVariant()` → variantId 不再由外部固定传入，改为由特征识别自动决定

### 需要新增的

- 人物分割模块（调用选定的分割模型）
- 区域颜色采样函数（5 区域独立采样 + 聚类）
- 特征识别函数（头发长度判断、服装类型判断、配饰检测）
- 发型绘制函数：`hairShort()` / `hairLong()` / `hairCurly()`
- 发型/服装选择逻辑
- 微调 UI（如果 P1 实现）

## 验收标准更新建议

基于新流程，M4 的验收标准需要更新：

| 检查项 | 期望结果 |
|--------|----------|
| Import 图片 | 选择照片后，系统进行人物分割 + 区域采样 + 特征识别 |
| 像素小人个性化 | 预览小人根据照片特征呈现不同发型、不同肤色、不同服装配色 |
| 不同照片不同小人 | 上传两张不同风格的照片，产出的小人在发型/肤色/服装上有明显差异 |
| 离线运行 | 断网状态下全流程正常（分割模型已本地缓存） |
| 占位角色 | 未上传照片前，显示 approved_base 默认小人 |
| Save 持久化 | 保存后重启仍激活自定义角色 |
| 交互状态正常 | 自定义角色的 poke/drag/sleep 状态仍正常 |
| 模型加载 | 首次 Import 加载分割模型 < 5s（有进度提示） |
| 错误处理 | 无效图片或分割失败时降级为旧流程（不崩溃） |

## 附录：千岛app参考流程映射

千岛app的拼豆岛功能是本次重设计的参考标杆，其核心流程映射到 Project Seal：

| 千岛步骤 | Project Seal 对应 |
|----------|-----------------|
| 上传图片 | Import 按钮选择本地照片 |
| 系统分析图片，提取人物特征 | 人物分割 + 5 区域颜色采样 + 特征识别 |
| 生成符合像素规范的图纸/小人 | 特征→像素映射规则表 + 模板部件组合 |
| 用户确认并保存 | 预览 + Save 按钮 |

**关键差异**：千岛产出的是"拼豆图纸"（静态图纸），Project Seal 产出的是"桌宠角色"（动态角色）。因此 Project Seal 的像素小人需要兼顾动画帧的兼容性（当前 MVP 只用单帧，但需预留动画帧扩展空间）。
